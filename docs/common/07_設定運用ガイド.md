# 設定運用ガイド

## 方針

設定は次の 2 層で管理します。

- `config/config.json.template`: 共有する基準値
- `config/config.json`: 各マシン固有の実値

この分離を崩すと、ローカルのドライブ名や IP アドレスがドキュメントや CI と乖離しやすくなります。

ツール別の指示テンプレートは次の 2 系統で管理します。

- 共通思想: `docs/common/11_自律開発コア.md`
- ツール固有テンプレート:
  - Claude Code: `scripts/templates/CLAUDE.md`
  - Codex CLI: `scripts/templates/AGENTS.md`
  - GitHub Copilot CLI: `scripts/templates/copilot-instructions.md`

## template に置くもの

- キー構造
- デフォルト値
- 各ツールの標準コマンド
- 共通の引数や環境変数の雛形

例:

- `tools.defaultTool`
- `tools.claude.command`
- `tools.codex.command`
- `tools.codex.args`
- `tools.copilot.command`
- `tools.copilot.args`

## config.json に置くもの

- ローカルのドライブパス
- UNC パス
- SSH 接続先
- 環境ごとの除外ディレクトリ
- 実際の API キー注入方法

代表例:

- `projectsDir`
- `sshProjectsDir`
- `projectsDirUnc`
- `linuxHost`
- `linuxBase`
- `localExcludes`

## 変更手順

1. 先に `config/config.json.template` を更新する
2. 必要なら `config/README.md` や関連ドキュメントを更新する
3. 各端末の `config/config.json` に差分を反映する
4. `Start-*.ps1 -DryRun` で影響を確認する

## Agent Teams backlog ルール

`Sync-AgentTeamsBacklog.ps1 -ApplyMetadata` は `config/agent-teams-backlog-rules.json` を読み取り、`TASKS.md` の自動抽出セクションへ `Priority / Owner / Source` を付けます。

- `config/agent-teams-backlog-rules.json.template`: 共有テンプレート
- `config/agent-teams-backlog-rules.json`: 実運用ルール

推奨の分担:

- `Ops`: MCP、runtime probe、共有ドライブ、worktree
- `QA`: diagnostics、tests、CI、verify
- `Architect`: Agent Teams、ClaudeOS、dashboard、lane
- `ScrumMaster`: docs、README、運用整理

## 起動スクリプトとの関係

現行の `Start-*.ps1` は `config/config.json` を読み込みますが、テストや CI では `AI_STARTUP_CONFIG_PATH` 環境変数で差し替えできます。

この仕組みにより、実機固有の設定を汚さずに自動検証ができます。

## ツール別テンプレートの扱い

- Claude Code は Agent Teams 前提の `CLAUDE.md`
- Codex CLI は manager-worker orchestration 前提の `AGENTS.md`
- GitHub Copilot CLI は custom agents + fleet 前提の `.github/copilot-instructions.md`

実行前に各ツールのグローバル設定をどのファイルへ反映するかは `docs/common/13_グローバル設定適用設計.md` を参照してください。

## Codex の起動基準

Codex は `codex --full-auto` をランチャー標準にします。

- 標準起動: `codex --full-auto`
- 追加プロファイル: `.codex/config.toml` の `default` / `full_auto` / `yolo`
- `yolo` profile: `approval_policy = "never"` と `sandbox_mode = "danger-full-access"` を使う限定用途

`yolo` profile は用意しますが、常用前提にはしません。

## Copilot の扱い

Copilot は `copilot --yolo` を前提に統一しています。

```json
"copilot": {
  "enabled": true,
  "command": "copilot",
  "args": ["--yolo"]
}
```

Copilot は `copilot --yolo` 基準で運用します。

## Cron HTML メールレポート設定 (v3.2.0)

`config/config.json` の `email` セクションでメール送信機能の挙動を制御します。**SMTP 認証情報は config.json には書かず、Linux 環境変数で管理する**設計です(誤コミット防止)。

```json
"email": {
  "enabled": false,
  "smtp": {
    "host": "smtp.gmail.com",
    "port": 587,
    "useStartTls": true,
    "userEnvVar": "CLAUDEOS_SMTP_USER",
    "passEnvVar": "CLAUDEOS_SMTP_PASS"
  },
  "from": "<your-email>",
  "to": "<your-email>",
  "subjectPrefix": "[ClaudeOS]",
  "scriptPath": "/home/kensan/.claudeos/report-and-mail.py",
  "enableEnvVar": "CLAUDEOS_EMAIL_ENABLED"
}
```

### Linux 環境変数(`~/.env-claudeos`、`chmod 600`)

| 変数 | 必須 | 用途 |
|---|:---:|---|
| `CLAUDEOS_SMTP_USER` | ✅ | Gmail アドレス(認証ユーザー) |
| `CLAUDEOS_SMTP_PASS` | ✅ | Gmail アプリパスワード |
| `CLAUDEOS_EMAIL_ENABLED` | ✅ | `1` で送信有効(既定 off / 誤送信防止) |
| `CLAUDEOS_DEFAULT_TO` | ⭕ | 省略時 `CLAUDEOS_SMTP_USER` |
| `CLAUDEOS_DEFAULT_FROM` | ⭕ | 省略時 `CLAUDEOS_SMTP_USER` |

詳細手順とトラブルシュートは [`16_HTMLメールレポート設定.md`](./16_HTMLメールレポート設定.md) と [`05_トラブルシューティング.md`](./05_トラブルシューティング.md#cron-html-メールレポート-v320) を参照。
